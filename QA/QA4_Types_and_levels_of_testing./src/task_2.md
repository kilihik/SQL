# Уровни тестирования программного обеспечения

## Таблица уровней тестирования

| Уровень тестирования | Что проверяем и на каком этапе жизненного цикла ПО | Кем проводится | Типичные артефакты для тестирования | Пример дефекта, характерного для этого уровня |
|---------------------|---------------------------------------------------|----------------|-------------------------------------|---------------------------------------------|
| **Модульное** | Проверяет отдельные компоненты, функции, методы. Проводится на ранних этапах разработки, сразу после написания кода. | Разработчики, реже - инженеры по тестированию | 1. Исходный код функций<br>2. Отдельные классы и методы<br>3. Юнит-тесты (unit tests)<br>4. Модули библиотек | 1. **Утечка памяти** в функции обработки изображений (Adobe Photoshop)<br>2. **Некорректное округление** в финансовых расчетах (банковское ПО)<br>3. **Ошибка граничных значений** в валидации email-адреса |
| **Интеграционное** | Проверяет взаимодействие между модулями, компонентами, системами. Проводится после модульного тестирования. | Инженеры по тестированию, разработчики | 1. API endpoints<br>2. Базы данных и их подключения<br>3. Микросервисы<br>4. Внешние интеграции (платежные системы, SMTP) | 1. **Некорректная кодировка** при интеграции с legacy-системой (миграция данных)<br>2. **Race condition** при одновременном обращении к общему ресурсу (онлайн-банкинг)<br>3. **Таймауты соединения** с внешним API (авиабилеты) |
| **Системное** | Проверяет систему в целом, соответствие требованиям. Проводится на почти готовом продукте. | Инженеры по тестированию, QA-инженеры | 1. Полностью собранное приложение<br>2. Тест-кейсы end-to-end<br>3. Функциональные требования<br>4. Пользовательские сценарии | 1. **Падение приложения** при высокой нагрузке (социальные сети)<br>2. **Некорректная работа в разных браузерах** (веб-приложения)<br>3. **Утечка конфиденциальных данных** через кэш (мессенджеры) |
| **Приемочное** | Проверяет готовность системы к использованию в production. Проводится перед выпуском. | Пользователи, бизнес-аналитики, заказчики | 1. Production-like среда<br>2. Бизнес-требования<br>3. User acceptance тесты<br>4. Документация для пользователей | 1. **Неудобный UI/UX** для целевой аудитории (CRM-система)<br>2. **Отсутствие необходимой функциональности** с точки зрения бизнеса (ERP-система)<br>3. **Несоответствие законодательным требованиям** (медицинское ПО) |
