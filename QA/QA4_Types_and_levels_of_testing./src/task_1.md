# Классификация тестирования мобильного банковского приложения
---
## Таблица категоризации тестирования

| По стадии проведения процесса тестирования | По знанию устройства системы | По направленности тестового сценария | По выполнению кода |
|---------------------------------------------|-------------------------------|--------------------------------------|---------------------|
| **Alfa-тестирование**: тестирование макета интерфейса (п.2) силами дизайнеров и продукт-менеджеров на удобство и логику переходов | **Black-Box**: тестирование модуля перевода средств без знания внутренней реализации - проверка корректности сумм, валидации ввода, сообщений об ошибках | **Позитивное тестирование**: проверка успешного перевода средств с корректными данными (правильная сумма, существующие счета) | **Статическое тестирование**: анализ кода модуля перевода средств без его выполнения (code review, проверка стандартов кодирования) |
| **Smoke-тестирование**: проверка базовой функциональности первой сборки приложения - запуск, авторизация, переход между основными экранами | **White-Box**: тестирование модуля перевода средств с доступом к коду - проверка алгоритмов расчета комиссий, обработки граничных значений | **Негативное тестирование**: проверка обработки некорректных данных при переводе (отрицательные суммы, несуществующие счета, превышение лимитов) | **Динамическое тестирование**: выполнение модуля перевода средств с различными тестовыми данными для проверки функциональности |
| **Unit-тестирование**: тестирование отдельных функций модуля перевода (валидация ввода, расчеты, форматирование) | **Gray-Box**: тестирование с частичным знанием внутренней структуры - проверка интеграции модуля перевода с UI без доступа к банковскому шлюзу | **Функциональное тестирование**: проверка соответствия модуля перевода функциональным требованиям (создание перевода, валидация, уведомления) | **Ручное тестирование**: исследовательское тестирование интерфейса приложения на различных устройствах |
| **Integration-тестирование**: проверка взаимодействия модуля перевода с другими компонентами приложения (база данных, UI-компоненты) | **Glass-Box**: полное тестирование с доступом ко всей внутренней структуре и коду приложения | **Нефункциональное тестирование**: проверка производительности интерфейса приложения (время отклика, плавность анимаций) | **Автоматизированное тестирование**: создание скриптов для регрессионного тестирования основных сценариев |
| **Sanity-тестирование**: быстрая проверка основных функций после сборки для внутренних сотрудников | **Closed-Box**: тестирование без какого-либо знания внутренней реализации системы | **Тестирование безопасности**: проверка защиты чувствительных данных в интерфейсе и при локальном хранении | **Статический анализ**: проверка кода на потенциальные уязвимости и ошибки без выполнения |

## Дополнительные примеры тестирования для текущего этапа
---
### По стадии проведения:
- **Компонентное тестирование**: тестирование модуля перевода средств изолированно от других компонентов
- **Приемочное тестирование**: проверка готовности функциональности для демонстрации заказчику
---
### По знанию системы:
- **Тестирование на основе моделей**: создание моделей поведения системы для генерации тестовых сценариев
- **Ad-hoc тестирование**: неформальное тестирование без подготовки тестовых случаев
---
### По направленности:
- **Регрессионное тестирование**: проверка, что новые изменения не сломали существующую функциональность
- **Usability тестирование**: оценка удобства использования интерфейса приложения
---
### По выполнению кода:
- **Инспекция кода**: формальный процесс проверки кода модуля перевода средств
- **Тестирование производительности**: оценка скорости работы интерфейса при различных условиях
