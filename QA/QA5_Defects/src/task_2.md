# 1. Тестирование в «стерильных» условиях
Суть проблемы: Систему проверяют только в идеальных условиях, не имитируя реальную эксплуатацию.

Примеры из практики:

Ariane 5: Унаследованный код не тестировали на реальных параметрах новой ракеты

iPhone 1970: Не проверяли последствия установки экстремальных значений даты

Что делать:

- Тестировать на максимально близком к оригинальному оборудовании

- Использовать реалистичные данные и нагрузки

- Проводить тестирование граничных значений и негативные тесты

# 2. Слепая вера в унаследованный код
Суть проблемы: Предположение, что старый код будет работать в новых условиях без дополнительной проверки.

Пример: Авария Ariane 5 — код с Ariane 4 использовали без адаптации к новой ракете.

Что делать:

- Проводить полное регрессионное тестирование для унаследованных компонентов

- Анализировать совместимость при интеграции старых и новых систем

- Использовать статический анализ кода для выявления потенциальных проблем

# 3. Хаотичное развертывание и настройка
Суть проблемы: Отсутствие контролируемых процессов переноса кода между средами.

Пример: Knight Capital — тестовый код попал в продакшен из-за ошибки развертывания.

Что делать:

- Внедрить автоматизированные пайплайны CI/CD

- Использовать Infrastructure as Code для идентичности сред

- Вести строгий контроль версий всего: кода, конфигураций, скриптов

# 4. Игнорирование системного взаимодействия
Суть проблемы: Компоненты тестируют изолированно, не проверяя их совместную работу под нагрузкой.

Примеры:

Ariane 5: Не проверили взаимодействие навигационных систем

Knight Capital: Не оценили влияние сбоя на всю торговую систему

Что делать:

- Увеличить долю интеграционного и системного тестирования

- Проводить нагрузочное тестирование всей системы

- Моделировать отказы компонентов и проверять отказоустойчивость

# 5. Предсказуемые тестовые сценарии
Суть проблемы: Тестировщики проверяют только ожидаемые сценарии, не учитывая нестандартное поведение пользователей.

Пример: Ошибка iPhone 1970 — никто не предположил, что пользователь установит такую дату.

Что делать:

- Привлекать к тестированию неподготовленных пользователей

- Проводить исследовательское тестирование

- Анализировать возможные сценарии злоупотребления системой

- Использовать техники тест-дизайна для покрытия неочевидных случаев

# Получается что серьезные дефекты проходят там, где тестирование становится формальностью. Эффективное тестирование требует:

Скептического мышления — постоянно задавать вопрос «а что если?»

Реалистичных условий — тестировать так, как система будет работать на самом деле

Контроля процессов — автоматизировать то, что можно автоматизировать

Системного подхода — видеть картину в целом, а не отдельные компоненты

Избегая этих антишаблонов, команды могут значительно снизить риск пропуска критических дефектов.
